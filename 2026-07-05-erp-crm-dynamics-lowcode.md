## 1. ERP vs CRM — 회사 업무 관리 소프트웨어

- **ERP (Enterprise Resource Planning, 전사적 자원 관리)**
  - 회사 **안**의 돈·물건·자원 관리: 회계, 재무, 재고, 구매, 생산, 공급망.
  - 예: "이번 달 매출?", "재고 몇 개?", "부품 언제 입고?"
- **CRM (Customer Relationship Management, 고객 관계 관리)**
  - 회사 **밖**의 고객·영업 관리: 영업 기회, 고객 정보, 마케팅, 고객 서비스.
  - 예: "이 고객과 언제 미팅?", "이 영업 건 성사 확률?"
- **한 줄 정리**: ERP = 회사 안(돈·물건) / CRM = 회사 밖(고객·영업).

### 왜 회사들이 이걸 사나
직접 만들면 몇 년+큰 비용. MS가 만들어둔 검증된 제품을 사서
자기 회사에 맞게 조금 고쳐 쓰는 게 빠르고 안전. 그 "고쳐 쓰는 작업"을 회사에서 대행.

## 2. Dynamics 365 — MS의 ERP+CRM 통합 제품

- 마이크로소프트가 만든 기업 업무 통합 관리 소프트웨어.
- ERP 기능 + CRM 기능을 하나로 묶은 제품.
- 회사가 사서 자기 업무에 맞게 커스터마이징해서 씀.
- 내부적으로 SQL Server / Dataverse 기반 → **DB 지식이 쓰인다.**
- 커스터마이징은 **C#/.NET 플러그인 + JavaScript**로 함 → 내 C# 스택 연결.

## 3. 로우코드 / 노코드 — 마우스로 조립하는 개발

- **로우코드**: 코드를 아주 조금만 쓰고, 화면에서 부품을 드래그&드롭으로
  끌어다 조립하는 개발 방식. 복잡한 로직만 짧은 수식(엑셀 함수 수준)으로 보완.
- **노코드**: 코드를 아예 안 씀.
- **MS의 로우코드 도구 = Power Platform**
  - Power Apps: 앱을 드래그로 제작
  - Power Automate: 반복 업무 자동화
  - Power BI: 데이터 시각화
  - Power Pages: 웹사이트 제작

### 오늘 헷갈렸다가 이해한 것 ①
> "로우코드 = 클로드에서 코드 복붙하는 거 아냐?"
- **아니었다.** AI 복붙은 여전히 "코드 텍스트"를 다루는 것.
- 로우코드는 애초에 코드가 거의 없고, 파워포인트에서 도형 그리듯
  **마우스로 부품을 조립**하는 것. 결과물이 코드 파일이 아니라 조립된 화면.

### 오늘 헷갈렸다가 이해한 것 ②
> "C#도 틀(프레임워크) 만들고 시작하는데, 로우코드랑 뭐가 달라?"
- **반은 맞다.** C#도 프로젝트 틀·라이브러리·스캐폴딩을 쓴다.
  둘 다 "이미 만들어진 걸 재사용"한다는 방향은 같음.
- **차이는 "추상화 수준"이다:**
  ```
  낮은 추상화 (세부사항 직접)
  │ 어셈블리
  │ C
  │ Java / C#  ← 틀 쓰지만 코드로 다 작성
  │ 로우코드    ← 코드를 드래그로 대체
  ▼ 노코드     ← 코드 완전히 숨김
  높은 추상화 (세부사항 숨김)
  ```
- C#은 "틀 위에 **코드를 글자로 쓰는**" 층.
  로우코드는 "코드 자체를 **마우스 조립으로 대체하는**" 한 단계 위 층.
- 즉 "틀을 쓴다"는 건 상대적. 로우코드는 C#보다 한 단계 더 높은 틀을 쓰는 것.

## 4. 추상화(abstraction) — 오늘의 핵심 개념

- 프로그래밍은 "세부사항을 얼마나 숨기느냐(추상화)"의 연속.
- **추상화가 높을수록 → 빠르지만 자유도 낮고, 대체되기 쉬움.**
- **추상화가 낮을수록 → 느리지만 강력하고, 대체 불가능.**
- 로우코드만 하는 사람 = 높은 추상화만 다룸 = 비개발자도 배울 수 있어 대체 쉬움.
- 밑바닥 코드(C#/Java)까지 아는 사람 = 로우코드 한계를 코드로 뚫음 = 대체 불가능.

### 면접용 한 문장
"Dynamics 365는 MS의 ERP·CRM 통합 솔루션이고, Power Platform은
로우코드로 앱·자동화를 빠르게 만드는 도구로 이해했습니다.
로우코드는 속도가 빠르지만 복잡한 로직은 결국 C# 같은 코드로 보완해야 하고,
제 C# 경험은 그 보완 부분에, DB 경험은 데이터 마이그레이션에 쓰일 수 있다고 봅니다."

## 5. 내 스택이 이 직무에 어떻게 쓰이나

| 내 스택 | ANH 컨설턴트 직무에서 |
|---|---|
| C# | Dynamics 플러그인 개발 (핵심) |
| Oracle/MySQL, DB 설계 | 데이터 마이그레이션·매핑 |
| JavaScript | Dynamics 화면 커스터마이징 |
| 영어 | 글로벌 제품 문서·해외 협업 |
| Java/Spring | 이 회사에선 거의 안 씀 (MS 스택) |

## 6. 오늘의 교훈

- 용어를 외우는 게 아니라 **"내가 어디서 헷갈렸고 어떻게 구분했는지"**가 진짜 학습.
- "로우코드 = 복붙?", "C#도 틀 쓰잖아?" 같은 의심을 던진 게
  추상화라는 핵심 개념까지 도달하게 함.
- **로우코드로 빠르게 만드는 능력 + 밑바닥 코드를 이해하는 능력, 둘 다 가진
  사람이 대체 불가능하다.**
