# 2026-06-08 TIL — Database Advance: Integrity, System Catalog & Transaction

어제 설계한 `hospital_reservation` DB 스키마 구현에 이어, 시스템의 안정성과 데이터 무결성을 보장하기 위한 핵심 CS(컴퓨터 사이언스) 이론을 학습하고 실무적 관점에서 복기함. 판교형 백엔드 엔지니어가 갖춰야 할 '데이터 신뢰성 제어 능력'의 밑바탕을 다지는 단계.

### 1. 데이터 무결성 (Data Integrity)의 실무적 적용
데이터베이스에 저장된 데이터의 정확성, 일관성, 유효성을 지키기 위한 필수 제약조건을 분석함.

* **개체 무결성 (Entity Integrity):** 모든 테이블은 기본키(PK)를 가져야 하며, PK는 NULL이 될 수 없고 중복될 수 없음. (예: `member_id`는 환자를 식별하는 절대적 기준이 됨)
* **참조 무결성 (Referential Integrity):** 외래키(FK) 값은 참조하는 부모 테이블의 PK 값과 일치하거나 NULL이어야 함. 환자 예약(`appointment`) 데이터는 반드시 존재하는 회원(`member`) 테이블의 PK만을 참조하도록 강제하여 데이터 고아 현상(Orphan Data)을 방지함.
* **도메인 무결성 (Domain Integrity):** 특정 컬럼의 값은 지정된 타입, 범위, 규칙을 만족해야 함. (예: 진료 예약일은 `DATETIME` 포맷을 준수해야 함)

### 2. 시스템 카탈로그 (System Catalog / Data Dictionary)의 이해
* 데이터베이스 자체의 메타데이터(테이블 구조, 인덱스 정보, 제약조건 등)가 저장되는 시스템 테이블.
* 사용자가 DDL(CREATE, ALTER 등)을 실행하면 DBMS가 자동으로 시스템 카탈로그를 갱신함. 
* 백엔드 엔지니어 관점에서 복잡한 병원 SI 시스템이나 쿼리 최적화(Optimization) 작업을 할 때, 시스템 카탈로그 정보를 조회하여 인덱스 설정 현황이나 테이블 스키마 구조를 역추적하고 디버깅하는 용도로 활용할 수 있음을 인지함.

### 3. 트랜잭션 (Transaction)과 ACID 원칙
병원 예약 및 결제 시스템 등 동시성 제어가 필수적인 비즈니스 로직에서 시스템 무너짐을 방지하는 핵심 개념. 데이터의 '전부(All) 또는 전무(Nothing)' 처리를 보장해야 함.

* **원자성 (Atomicity):** 트랜잭션 내의 연산은 전체가 완료(Commit)되거나 전체가 취소(Rollback)되어야 함. 진료 예약은 성공했는데 예약 로그 기록에 실패했다면 전체 프로세스를 Rollback 해야 함.
* **일관성 (Consistency):** 트랜잭션 수행 전후의 데이터베이스는 항상 모순 없는 일관된 상태(무결성 제약조건 충족)를 유지해야 함.
* **고립성 (Isolation):** 동시에 실행되는 여러 트랜잭션이 서로의 연산에 끼어들지 못하도록 격리해야 함. (하나의 의사 타임슬롯에 두 명의 환자가 동시에 예약되는 동시성 리스크 제어의 기반)
* **영속성 (Durability):** 성공적으로 완료된 트랜잭션의 결과는 시스템에 장애가 발생하더라도 영구적으로 보존되어야 함.

### 4. Next Action & 판교형 관점
* 오늘 학습한 **트랜잭션(ACID)** 개념을 바탕으로, MyBatis 환경에서 `@Transactional` 어노테이션을 활용해 병원 예약 로직의 예외 처리(Exception Handling) 프로토콜을 설계할 예정.
* 다수의 사용자가 동시에 예약 요청을 보낼 때 발생할 수 있는 동시성 이슈를 차단하고, **Swagger/Postman**을 통해 프론트엔드와 트랜잭션 실패 시의 에러 코드(Error Code) 규격을 명확히 공유하여 소통 비용을 최소화할 것임.
